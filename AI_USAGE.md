# AI 图片筛选与评审工作台 - AI 使用说明

## 1. 文档目的

本文件用于说明本项目在产品分析、技术架构、代码实现与文档撰写过程中，如何合理使用 AI 作为辅助工具。

核心立场：AI 是加速器，不是决策者。它可以帮助提升调研、组织与重复性工作的效率，但产品判断、技术取舍、优先级排序和最终交付责任仍由我自己承担。

## 2. 使用 AI 的环节

| 环节        | 使用方式                                                      | 工具              |
| ----------- | ------------------------------------------------------------- | ----------------- |
| 产品分析    | 让 AI 帮我列举图片评审场景的痛点，我基于经验筛选和补充        | Chat + Qwen3            |
| 技术架构    | 和 AI 讨论 Electron + .NET 的集成方案，确认技术可行性         | Chat + DeepSeek      |
| 代码编写    | 用 AI 理解需求和核心功能设计并辅助生成代码       | GitHub Copilot + Claude Sonnet 4.6   |
| 文档撰写    | 本方案各文档的结构组织参考了 AI 建议，内容由我自己判断与修改    | GitHub Copilot + GPT 5.4   |

## 3. 各环节中的具体使用方式

### 3.1 产品分析

在需求澄清阶段，我使用 AI 帮助列举图片评审场景中常见的问题，例如：

- 批量图片难以快速初筛。
- 人工记录状态和备注不统一。
- 交付时需要重新整理入选结果。
- 两张接近图片难以高效比较。

AI 在这一环节主要承担发散与补充作用，帮助我更快展开问题空间。但哪些问题值得纳入当前版本，仍由我根据业务经验与交付时间决定。

### 3.2 技术架构

在后端数据库选择时，AI建议使用更轻量的SQLite，而不是我擅长的SQL Server。在确认ABP框架支持多种数据库提供程序，果断采用AI的建议使用SQLite而非SQL Server。
在桌面化方案评估时，我与 AI 讨论了 Electron、Tauri 以及纯 Web 交付的可行性，重点确认：

- Vue 前端能否以静态资源方式被 Electron 加载。
- .NET 后端能否在 Electron 启动时被主进程拉起。
- 本地端口通信是否可满足现有业务 API。
- Windows 安装包构建链是否可快速落地。

AI 提供的是方案备选与风险提示，最终决定采用更成熟的 Electron，是基于我对 .NET 技术栈更熟悉、时间有限且需要快速交付的现实判断。

### 3.3 代码编写

在代码实现中，我会使用 AI 辅助完成重复性较高、模式相对固定的部分，例如：

- DTO 定义。
- CRUD 接口基础结构。
- Mapper 或映射类样板代码。
- 前端页面初始骨架。

但对于核心业务逻辑，例如：

- 后端系统架构的搭建，编写测试服务让AI准确理解ABP最新版本的框架。
- 后端统一的接口返回格式（导出图片接口除外）、异常过滤器、异常中间件等。
- Electron 主进程启动后端。
- 桌面端请求链路适配。

我会自己梳理逻辑、确认实现方式并进行实际验证，不直接依赖 AI 一次性生成。

### 3.4 文档撰写

本项目的设计文档、打包说明和用户手册在结构组织上参考了 AI 的内容和建议。AI提供初版，我结合项目事实、技术实现、取舍原因相关的内容确认修改，避免出现“结构完整但内容失真”的问题。

## 4. 如何判断 AI 建议是否采纳

### 4.1 技术选型建议

AI 强烈建议数据库提供选择SQLite，考虑技术完全可行且更轻量，第一天就实战切换为SQLite，经过实践确认可行。
Electron 桌面容器方案的可行性，我和多个AI 沟通过，在GitHub 也能看到成功案例，最终确认可行。
部分AI 建议使用 Tauri 替代 Electron，因为它更轻量、安装包更小。但考虑到我对 .NET 生态更熟、3 天时间有限，而且当前目标是先稳定完成桌面化交付，所以我选择了更保险的 Electron 方案。

判断逻辑不是“哪个更新”，而是“哪个更适合当前目标与时间约束”。

### 4.2 功能建议

AI 曾建议加入“AI 自动分类”功能，这在长期规划中是合理方向，但对于当前原型版本来说，优先级明显低于手动快捷标记与导出交付。因此我将其放到 P2，而不是在首版中强行加入。

## 5. 我的采纳原则

在本项目中，我对 AI 建议的采纳遵循以下原则：

### 原则一：是否符合当前版本目标

如果建议会让产品偏离“快速交付一个可运行原型”的目标，就不采纳或延后。

### 原则二：是否能被验证

任何涉及技术实现、性能、稳定性、兼容性的建议，必须经过代码验证或样本验证，不能只因为回答看起来完整就直接采用。

### 原则三：是否降低而不是增加复杂度

AI 给出的方案如果会引入额外复杂度，例如更多依赖、额外部署步骤、较高维护成本，即使理论更先进，也不一定适合当前项目。

### 原则四：最终责任必须清晰

所有进入项目的设计、代码和文档，都需要我自己理解、确认并负责。不能把不确定的内容直接归因于 AI。

## 6. 本项目中 AI 不负责的内容

- 产品优先级排序。
- 技术路线最终拍板。
- 核心业务逻辑与边界处理。
- 最终代码质量与测试运行结果验证。

## 7. 总结

本项目中的 AI 使用方式强调三点：

1. 用 AI 提高信息整理、重复性工作和编写常规逻辑代码的效率。
2. 用人工判断控制方案质量和方向。
3. 用实际验证确保结果不是“看起来合理”，而是真正可运行、可交付。

因此，AI 在这个项目里承担的是辅助角色，而不是主导角色。
